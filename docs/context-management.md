# Context Management

> How Claude Code manages conversation context, prevents token overflow, and compresses history.

---

## Overview

Large language models operate within a finite **context window** â€” the maximum number of tokens that can be sent in a single API request. As conversations grow, accumulated messages (user input, assistant responses, tool calls, tool results) consume this window until the model can no longer accept new input.

Claude Code addresses this with a layered context management pipeline that runs on every query turn. Each layer targets a different cost/quality tradeoff, from lightweight tool-result pruning to full conversation summarization.

---

## Context Pipeline

The context management pipeline runs inside the query loop (`src/query.ts`) before each API call. The stages execute in a fixed order:

```
                          Query Loop Entry
                               |
                    +----------v-----------+
                    |  Content Replacement  |  Deduplicate large tool outputs
                    +----------+-----------+
                               |
                    +----------v-----------+
                    |        Snip          |  Remove marked message ranges
                    |  (HISTORY_SNIP flag) |  (feature-gated, ant-only)
                    +----------+-----------+
                               |
                    +----------v-----------+
                    |    Microcompact      |  Clear old tool results
                    |  (time-based or      |  to reclaim tokens without
                    |   cache-editing)     |  a full summarization
                    +----------+-----------+
                               |
                    +----------v-----------+
                    |  Context Collapse    |  Commit older context to
                    |  (CONTEXT_COLLAPSE)  |  a separate store
                    +----------+-----------+
                               |
                    +----------v-----------+
                    |    Autocompact       |  Full conversation summary
                    |  (threshold-based)   |  via forked LLM call
                    +----------+-----------+
                               |
                    +----------v-----------+
                    |   Blocking Limit     |  Hard stop if still over
                    |      Check           |  the context window
                    +----------+-----------+
                               |
                          API Call
```

Each stage is independent â€” earlier stages may reduce token counts enough that later stages become no-ops. The design deliberately runs cheaper operations first (snip, microcompact) before resorting to expensive summarization (autocompact).

## Autocompact

**Location:** `src/services/compact/autoCompact.ts`, `src/services/compact/compact.ts`

Autocompact is the primary context compression mechanism. It replaces the entire conversation history with a structured summary generated by a forked LLM call.

### How It Works

1. **Threshold check:** After each turn, `shouldAutoCompact()` estimates the current token count using `tokenCountWithEstimation()` and compares it against a threshold derived from the model's effective context window minus a buffer:

   ```
   threshold = effectiveContextWindow - AUTOCOMPACT_BUFFER_TOKENS (13,000)
   ```

2. **Session memory compaction (first attempt):** Before falling back to full summarization, the system tries `trySessionMemoryCompaction()` â€” a lighter-weight approach that uses the Session Memory subsystem's extracted knowledge to reconstruct a compact context, preserving recent messages verbatim.

3. **Full summarization (fallback):** If session memory compaction is unavailable, `compactConversation()` sends the entire message history to a forked LLM call with a detailed summarization prompt. The prompt instructs the model to produce a structured summary covering:
   - Primary request and intent
   - Key technical concepts
   - Files and code sections (with snippets)
   - Errors and fixes
   - All user messages
   - Pending tasks and current work

4. **Post-compact assembly:** The result replaces all prior messages with:
   - A compact boundary marker (system message)
   - The summary (as a user message)
   - File attachments for recently-read files (up to 5 files, 50K token budget)
   - Plan attachments (if an active plan exists)
   - Post-compact hook results

5. **Cleanup:** Caches are invalidated (`runPostCompactCleanup()`), microcompact state is reset, and CLAUDE.md is re-read to catch any changes since the conversation started.

### Circuit Breaker

If autocompact fails 3 consecutive times (`MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES`), the system stops retrying for the remainder of the session. This prevents wasting API calls when the context is irrecoverably over the limit.

### Prompt-Too-Long Retry

If the compaction API call itself exceeds the context window, `truncateHeadForPTLRetry()` drops the oldest message groups and retries (up to 3 times). This is a lossy but unblocking escape hatch.

### Configuration

| Mechanism | Description |
|-----------|-------------|
| `autoCompactEnabled` in user config | Master switch (default: `true`) |
| `DISABLE_COMPACT` env var | Disables all compaction |
| `DISABLE_AUTO_COMPACT` env var | Disables auto-compact only (manual `/compact` still works) |
| `CLAUDE_CODE_AUTO_COMPACT_WINDOW` env var | Override the effective context window size |
| `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` env var | Set threshold as a percentage of context window |

### The `/compact` Command

Users can trigger compaction manually via `/compact [instructions]`. This follows the same pipeline (session memory first, then full summarization) but allows optional custom instructions that are appended to the summarization prompt. Example:

```
/compact Focus on the TypeScript migration and keep all code snippets
```

The command is defined in `src/commands/compact/compact.ts`.

## Microcompact

**Location:** `src/services/compact/microCompact.ts`

Microcompact is a lighter-weight alternative to autocompact that selectively clears old tool results instead of summarizing the entire conversation. It preserves conversation structure while reclaiming tokens from verbose tool outputs.

### Compactable Tools

Only tool results from specific tools are eligible for clearing:

- `Read` (file read)
- `Bash` / shell tools
- `Grep`, `Glob`
- `WebSearch`, `WebFetch`
- `Edit` (file edit)
- `Write` (file write)

### Two Microcompact Paths

#### 1. Time-Based Microcompact

**When:** The gap between now and the last assistant message exceeds a configurable threshold (default: 60 minutes). This signals that the server-side prompt cache has expired, so the full prefix will be rewritten anyway.

**What it does:** Content-clears all but the N most recent compactable tool results (default: keep last 5), replacing their content with `[Old tool result content cleared]`.

**Why:** Since the cache is cold, clearing old results before the request shrinks what gets rewritten â€” there is no cache-miss penalty because the miss would happen regardless.

Configuration is loaded from GrowthBook (`tengu_slate_heron`):
- `enabled`: master switch (default: `false`)
- `gapThresholdMinutes`: trigger threshold (default: 60)
- `keepRecent`: number of recent tool results to preserve (default: 5)

#### 2. Cached Microcompact (Cache Editing)

**When:** The prompt cache is warm and the number of tracked compactable tool results exceeds a configured trigger threshold.

**What it does:** Instead of mutating message content, it uses the API's `cache_edits` mechanism to delete tool results from the server-side cache without invalidating the cached prefix. The local messages remain unchanged â€” `cache_reference` and `cache_edits` blocks are added at the API layer.

**Why:** This avoids the cache-miss penalty of content mutation. Deleting from the warm cache is cheaper than rewriting the entire prefix.

This path is gated behind the `CACHED_MICROCOMPACT` feature flag and requires model support for cache editing.

### Execution Order

Microcompact runs **after** snip but **before** autocompact. If microcompact frees enough tokens, autocompact may not need to fire.

## Snip

**Location:** Referenced via `src/services/compact/snipCompact.ts` (feature-gated, not in open-source build)

Snip is a message-range removal mechanism gated behind the `HISTORY_SNIP` feature flag. Unlike autocompact (which summarizes) or microcompact (which clears tool results), snip **removes entire message ranges** from the conversation.

### How It Works

1. **Model-driven:** The `SnipTool` (loaded from `src/tools/SnipTool/`) is available to the model as a regular tool. When the model determines that a section of conversation history is no longer relevant, it can invoke the snip tool.

2. **Pre-query processing:** Before each API call, `snipCompactIfNeeded()` processes any pending snip operations, removing the targeted messages and producing a boundary message.

3. **Token accounting:** The `tokensFreed` value from snip is passed to autocompact's threshold check via `snipTokensFreed`, since `tokenCountWithEstimation()` reads token usage from the surviving assistant message which still reflects pre-snip context size.

4. **Projection:** `projectSnippedView()` filters snipped messages from the model-facing view while keeping them in the REPL's full history for UI scrollback.

### Context Efficiency Nudge

When `HISTORY_SNIP` is enabled, a `context_efficiency` attachment is injected into the conversation, nudging the model to use the snip tool proactively when it detects that older context is no longer needed.

### Force Snip Command

A `/force-snip` command (also feature-gated) allows users to manually trigger snip operations.

## Token Estimation

**Location:** `src/services/tokenEstimation.ts`

Token estimation is critical because the system must decide whether to trigger compaction *before* making an API call, not after.

### Estimation Methods

| Method | Accuracy | Cost | Use Case |
|--------|----------|------|----------|
| `roughTokenCountEstimation()` | Low (char_count / 4) | Free | Quick estimates, thresholds |
| `countTokensWithAPI()` | Exact | 1 API call | Single content strings |
| `countMessagesTokensWithAPI()` | Exact | 1 API call | Full message arrays |
| `countTokensViaHaikuFallback()` | Near-exact | 1 API call (Haiku) | When main model count unavailable |

### Rough Estimation Details

The default rough estimator divides byte length by 4 (approximately 4 bytes per token for English text). File-type-aware estimation uses different ratios:

- JSON files: 2 bytes per token (dense single-character tokens like `{`, `}`, `:`)
- All other files: 4 bytes per token (default)

Microcompact's `estimateMessageTokens()` applies a 4/3 padding multiplier to rough estimates to be conservative â€” underestimating would risk letting oversized content through.

### Cache-Aware Counting

The `countTokensViaHaikuFallback()` function returns the sum of `input_tokens + cache_creation_input_tokens + cache_read_input_tokens`, giving a total that accounts for cached content.

## Context Assembly

**Location:** `src/context.ts`, `src/utils/systemPrompt.ts`, `src/constants/prompts.ts`

Before each API call, the full context is assembled from multiple sources:

```
+-----------------------------------------+
|          System Prompt                  |
|  (src/constants/prompts.ts)             |
|  - Base instructions                    |
|  - Tool-use guidelines                  |
|  - Safety instructions                  |
+-----------------------------------------+
|          System Context                 |
|  (src/context.ts :: getSystemContext)   |
|  - Git status (branch, recent commits)  |
|  - Cache breaker (if set)               |
+-----------------------------------------+
|          User Context                   |
|  (src/context.ts :: getUserContext)      |
|  - CLAUDE.md files (project, user,      |
|    enterprise)                          |
|  - Current date                         |
+-----------------------------------------+
|          Attachments                    |
|  (src/utils/attachments.ts)             |
|  - File content (from Read tool)        |
|  - Skill content (invoked skills)       |
|  - MCP instructions                     |
|  - Context efficiency hints             |
+-----------------------------------------+
|          Tool Definitions               |
|  (from registered Tool objects)         |
|  - Schemas, descriptions                |
+-----------------------------------------+
|          Message History                |
|  (after compact boundary, if any)       |
|  - User messages                        |
|  - Assistant messages                   |
|  - Tool use / tool result blocks        |
+-----------------------------------------+
```

Both `getSystemContext()` and `getUserContext()` are **memoized** â€” they are computed once at session start and cached for the conversation's lifetime. Post-compaction cleanup clears the `getUserContext` cache so CLAUDE.md changes are picked up on the next turn.

## Cost Tracking

**Location:** `src/cost-tracker.ts`, `src/bootstrap/state.ts`

Claude Code tracks token usage and cost across all API calls in a session, including compaction calls.

### What Is Tracked

| Metric | Description |
|--------|-------------|
| `input_tokens` | Tokens sent to the API |
| `output_tokens` | Tokens generated by the model |
| `cache_creation_input_tokens` | Tokens written to the prompt cache |
| `cache_read_input_tokens` | Tokens read from the prompt cache |
| Per-model breakdown | Usage split by model (main model, Haiku for token counting, etc.) |
| API duration | Total and per-retry wall-clock time |
| Lines changed | Lines added/removed by file editing tools |

### User Access

The `/cost` command displays a summary of token usage and estimated USD cost for the current session. Cost is calculated using model-specific pricing via `calculateUSDCost()` in `src/utils/modelCost.ts`.
